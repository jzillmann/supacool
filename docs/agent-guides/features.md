# Feature index — what exists and where it lives

One dense page mapping every shipped subsystem to its primary files, so nobody (human or
agent) has to rediscover a feature from source. **Maintenance rule**: when you ship a
feature or materially reshape one, update its row here in the same commit (see
`AGENTS.md` § Documentation system). Names are real file names — `fd <name>` finds them.

| Server lifecycle | Per-repo status/start/stop scripts (configured in Repository Settings, persisted to `<repo>/supacool.json`) driving a card + toolbar chip that starts/stops the workspace's dev server, with auto-stop on park/remove and auto-start on unpark. Script output is ANSI-stripped, then **scanned for ports**: the chip links the one worth opening and lists the rest in a context menu. The pick is a ranking, not a guess — a declared `http(s)://` URL wins, else a web-like port (3000–3999, 4200, 5173, 8000, 8080), else a lone port whatever its number; several unremarkable ports produce a menu rather than a wrong link. Only status/start output is scanned (a stop run would resurrect the ports it just took down), and a `.stopped` server offers no links even though its ports stay remembered for the next start. **Running vs stopped is not taken from the status script's exit code** — a `dev status` reports its fleet and exits 0 whether anything is up or not — so when the output names ports, the verdict comes from actually TCP-probing them (`PortReachabilityClient`): running only if one accepts a connection, else stopped. A status that names no ports falls back to the exit code. | `Supacool/Clients/ServerLifecycleClient.swift` (runner + `strippingANSIEscapeCodes`), `Supacool/Clients/PortReachabilityClient.swift` (TCP reachability), `Supacool/Domain/ServerEndpoint.swift` (`ServerEndpointScanner`), `ServerEndpointLinkChip.swift`, `BoardFeature.swift` (`ServerLifecycleViewState`, `serverLifecycle*` handlers, `resolvedStatus`), `RepositorySettings.swift` (`ServerLifecycleSettings`), `RepositorySettingsView.swift` | — |
